## This run: closed audit P0 item 4 — refuse to render empty/fallback prompts

**Found (already flagged 07-18, audit finding P0-4):** `promptService.generateImagePrompt` returned a hardcoded placeholder — `` `An empty page, ${style}` `` — for any page whose extracted text was empty or whitespace-only. This is indistinguishable, downstream, from a real illustration: the placeholder still goes through the full LLM-mediated rendering path and produces a real generated image. So a scanned/image-only PDF page with no text layer (currently a live gap — see 07-18's report on `extractTextFromImage` having zero working call sites) would silently render generic, content-free art instead of surfacing that nothing was actually read from that page.

**Fix:** `generateImagePrompt` now throws a new `EmptyPageError` (exported from `promptService.ts`) instead of returning the placeholder. No LLM call is made either way — empty input was already short-circuited before the network call, so this is a pure behavior change at the same check, not a new code path.

**Caller behavior (no changes needed):** both page-mode pipelines already wrap the `generateImagePrompt` call inside their existing outer `try/catch`:
- `processPagePipeline` (single-page, invoked by every automatic retry): outer catch records `processingStatus: "error"` on the page and rethrows. It does **not** call `markPageForRetry` — that only happens in the separate inner catch scoped to image-generation failures.
- `processPagePipelineWithContext` (main multi-page path): same shape — outer catch records the error, does not schedule retry, and `processBookPipeline`'s per-page loop treats it as one failed page (`failureCount++`) without aborting the rest of the book.

This means the fix required zero pipeline-layer changes — the pre-existing error-handling shape already implements the correct policy (fail permanently, don't retry) for exactly this kind of non-transient failure. Verified explicitly with new tests rather than assumed.

**Scene-mode unaffected by design:** `scenePlanner.selectScenes` filters pages to `>= 20` meaningful characters before ever calling the LLM, and `sanitiseLLMScenes`/`buildFallbackScenes` both discard any scene with an empty `description`. So `generateScenePrompts` never calls `generateImagePrompt` with empty text — confirmed by reading the code, not just assumed; no test changes were needed on the scene-mode side.

**Decoupling invariant upheld:** this is a prompt/render-side change only. `EmptyPageError`'s check happens before any OCR/transcription code is touched; the story bible and transcription paths are untouched.

**Tests:**
- `storyBibleBoundary.test.ts`: rewrote the one test that asserted the old placeholder behavior to assert `EmptyPageError` is thrown instead (both for `""` and whitespace-only input), with no LLM call. Added one new integration-level test using the real `pipelineService` + `promptService` (only `invokeLLM`/db/storage/image-gen mocked): a 3-page book where page 2 has no text — asserts `{ successCount: 2, failureCount: 1 }`, exactly 2 `generateImage` calls (never one for the blank page), the failed page recorded as `processingStatus: "error"`, `markPageForRetry` never called for it, and the book still reaches `"completed"` overall.
- `pipelineService.singlePage.test.ts`: added a unit test for the same behavior on the single-page retry path, importing the real `EmptyPageError` class via `vi.importActual` (the module mock stubs `generateImagePrompt` but now preserves the real error class) to simulate the rejection and assert no retry gets scheduled.

## Environment notes (for future runs)

- **Background processes still do not survive across separate shell-tool calls.** Re-confirmed this run: `npx tsc ... > log 2>&1 & disown` from one call, checked from the next call, was simply gone — only the two `socat` proxy processes (unrelated infra) remained. This matches 07-18's finding and rules out `nohup`/`setsid`/backgrounding as a workaround for anything that runs longer than the 45s call cap; a slow command must be retried as a plain foreground command until one attempt happens to land inside the window.

What made it tractable at all: the `tsconfig.json` `incremental: true` + `tsBuildInfoFile` cache (`node_modules/typescript/tsbuildinfo`) was still warm from 07-18's run, and since this session's only production-scope file change was `server/promptService.ts` (both test files touched are excluded from `npm run check` by its `**/*.test.ts` exclude), the incremental pass only had to recheck that one file and its direct dependents (`pipelineService.ts`, `scenePlanner.ts`). Future sessions: don't delete/rebuild `node_modules/typescript/tsbuildinfo` casually, and expect to retry `npm run check` 1-3 times rather than treating a single timeout as a real failure.
