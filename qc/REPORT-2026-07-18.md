## This run: fixed a real correctness bug in the single-page retry pipeline

**Found:** `processPagePipeline` (the from-scratch pipeline `retryWorker.ts` invokes on every automatic retry — confirmed the only caller) derived page text by running Tesseract OCR (`extractTextFromImage`) against `generatePageThumbnail`'s output. `generatePageThumbnail` returns a hardcoded 1×1 white PNG — real thumbnails are deliberately generated client-side to avoid a native `canvas` dependency (documented in `pdfService.ts`; confirmed `canvas` is not installed). So every Tesseract call on this path ran against a blank pixel and could only ever return empty/near-empty text — and since this is the retry path, it would silently **overwrite a page's real, previously-extracted text with empty text** on every retry, which then feeds `promptService`'s empty-text fallback ("An empty page, {style}") into image generation. This matches the audit reconciliation's H1 finding (`qc/AUDIT-RECONCILIATION-2026-07-17.md`) and explains why retries of image-gen-only failures were silently degrading output quality instead of just re-trying the image call.

The main multi-page pipeline (`processBookPipeline` → `processPagePipelineWithContext`) never had this problem — it derives page text from `extractPDFPages` (real pdfjs text-layer extraction) once per book and passes it in; it never calls Tesseract at all (confirmed by existing `core.test.ts` assertion `expect(mOcr).not.toHaveBeenCalled()`).

**Fix:** added `extractSinglePageText(pdfBuffer, pageNumber)` to `pdfService.ts` — the same pdfjs text-layer extraction `extractPDFPages` uses, scoped to one page so a retried page doesn't re-parse the whole document. `processPagePipeline` now calls this instead of running Tesseract against the stub thumbnail. This makes the retry path derive text the same way the main path does — consistent, and no longer capable of silently blanking a page's text.

**Un-wired (not deleted) as a consequence:** yesterday's `getOcrTextCached` (wired into this exact call site 2026-07-17 to cache Tesseract results) is removed from `pipelineService.ts`. Caching a guaranteed-wrong result doesn't help, and leaving the function defined-but-uncalled would be an inert landmine — its premise (that `generatePageThumbnail`'s output is a real image) doesn't hold, so any future call site copy-pasting it back in would silently reintroduce this exact bug. `ocrCacheService.ts` (the generic TTL cache) and its own unit tests are untouched — still sound, reusable infra for a real Tesseract fallback once server-side page rasterization actually exists. `server/pipelineService.ocrCache.test.ts` (which tested the removed wrapper) is repurposed in place as a regression guard asserting `getOcrTextCached` is gone — see Environment notes for why it's repurposed rather than deleted.

**Decoupling invariant upheld:** this is transcription-side only. `extractSinglePageText` touches pdfjs text extraction only — no import of, or dependency on, image generation, the prompt service, or the story bible. The new `server/pipelineService.singlePage.test.ts` includes an explicit render-boundary test (`generateImage` receives `prompt`/`keyPrefix`/`params` only, page text never leaks into the call) mirroring the existing pattern in `scenePipeline.test.ts`/`storyBibleBoundary.test.ts`.

**Consequence worth flagging:** `ocrService.ts`'s `extractTextFromImage` (Tesseract, the `tesseract.js` dependency) now has **zero live production call sites** anywhere in the app (confirmed by grep — only test files assert it's *not* called, plus the already-dead-per-audit `pipelineServiceOptimized.ts`). It was never actually functional in production before this fix either (guaranteed-blank input), so this isn't a regression, but it means: right now, there is no working path from "scanned/image-only PDF page" to extracted text anywhere in the app. `extractPDFPages`'s pdfjs text layer returns empty for such pages, `promptService` currently renders a generic "empty page" fallback prompt instead of failing (audit's H1/P0-item-4, still open, out of scope for this fix), and nothing attempts real OCR. Real Tesseract OCR can't do anything useful until there's an actual rendered page image to feed it — i.e. until server-side thumbnail/rasterization is implemented for real (currently deferred client-side; would need a `canvas` dependency or an alternative rasterizer). This is a product/architecture call, not a bug fix — flagging per standing instruction to surface anything the todo.md checkmarks overstate.

## Environment notes (for future runs)

- **New file could not be deleted; two files could not be deleted, one repurposed, one kept anyway.** Both plain `rm`, `mv`, and Node's `fs.unlinkSync` returned `EPERM` for `server/pipelineService.ocrCache.test.ts` — same class of issue as the long-documented unremovable `.git/*.lock` files, except this time on an ordinary tracked file, and it recurred on a **brand-new scratch file** (`tsconfig.qc-scratch.json`, created fresh this session) after a retry and a sleep. This confirms the restriction isn't about file age/recent access — it's an unpredictable, path-specific EPERM on this mount. Workarounds: the obsolete OCR-cache test file was repurposed in place (its Tesseract-cache tests removed since the function they tested no longer exists; replaced with a single regression-guard test plus a comment explaining the history — see file). The scratch tsconfig used for this report's test-inclusive typecheck (`tsconfig.qc-scratch.json`) could not be cleaned up afterward either, so it's left committed at the repo root rather than pretending it doesn't exist — it's a legitimate, reusable version of the same workaround the 2026-07-13 and 2026-07-16 sessions each recreated from scratch (root `tsconfig.json` excludes `**/*.test.ts` and its `types: ["node", "vite/client"]` can't resolve `vite/client` when the client is out of scope; the scratch config scopes to `server/+shared/`, includes tests, and overrides `types`/`typeRoots` to just `node`). Future sessions can reuse it directly: `npx tsc --noEmit -p tsconfig.qc-scratch.json`.
- **Long-running commands need to redirect to a log file, not rely on backgrounding.** The shell tool caps each call at 45s. A command that's still running when the cap hits gets killed — plain `&`-backgrounding, `nohup`, and `setsid`+`disown` were all tested this session and none survive the call boundary (confirmed via a `sleep 5 &` marker-file test finding nothing after 8s in a follow-up call). The reliable pattern: run the long command as the sole foreground command redirecting to a log file (`cmd > /tmp/x.log 2>&1; echo EXIT:$? >> /tmp/x.log`), accept the timeout error on commands that run past 45s, then poll the log file with quick, separate calls. `npx tsc --noEmit` (full project) and small vitest batches (1-2 files) reliably finish inside 45s; the full 16-file vitest suite does not — it needs chunking (2-6 files per call depending on whether they import the full `appRouter`/router tree, which adds ~10s+ of module-collection time per call regardless of test count).
- Stale `.git/HEAD.lock` / `.git/index.lock` / per-branch `refs/heads/*.lock` are still unremovable from the sandbox, same as every prior session. `git reset` (mixed, no path args) still works despite the lock warnings printed to stderr — used this session to clear a stale half-reverted index left over from a previous session's temp-index commit plumbing (verified byte-for-byte against `HEAD` before resetting; no working-tree content was at risk). Committing still needs the temp-index + `write-tree` + `commit-tree` + direct `refs/heads/<branch>` file overwrite workaround.
