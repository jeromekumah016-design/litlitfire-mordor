## This run: closed the `core.test.ts` type-debt item flagged 2026-07-13

Root cause (confirmed, not just papered over): `booksRouter.ts` has an in-memory query cache with a generic helper `getFromCache<T>(key): T | null`. Two procedures (`list`, `getDetails`) call it as `getFromCache(cacheKey)` with no explicit type argument and no inference site, so `T` defaults to `unknown`. The very next line, `if (cached) return cached;`, truthiness-narrows that `unknown` down to TypeScript's `{}` (the standard "definitely non-nullish, otherwise untyped" narrowing result) — which has none of the real fields. Since the procedure has a second `return result` later in the same function, the resolver's *inferred* return type becomes a union of `{}` and the real shape. `npm run check` never caught this because the root tsconfig excludes `**/*.test.ts`; it only surfaced when `core.test.ts` dereferenced fields like `result.pages`, `result.items`, `result.totalPrice` on values coming from `caller.books.list(...)` / `caller.books.getDetails(...)`.

Fix: added explicit result-shape types (`BooksListResult`, `BookDetailsResult`, plus their item types) in `booksRouter.ts` and bound them at both call sites — `getFromCache<BooksListResult>(...)` / `getFromCache<BookDetailsResult>(...)` — and annotated the `result`/`pageItems` locals with the same types. Both branches of each procedure (cache-hit and freshly-computed) now agree on one concrete type, so the union-with-`{}` collapse can't happen. Zero runtime change — this is types-only; the cache Map, TTL, and eviction logic are untouched.

Scope note: the same `getFromCache(cacheKey)` (no type arg) pattern still exists in `getDashboardStats`, `getLibraryOverview`, and `getProcessingMetrics`. Currently harmless (no test dereferences fields on their results, so the `{}` collapse stays invisible), but it's the same latent issue. Left alone this run per "advance one logical unit, don't sprawl" — flagging for a future pass if those procedures grow test coverage that touches specific fields.

## Environment notes (for future runs)

- **Refined diagnosis of the recurring file-tail-truncation bug**: this run caught it in the act. After 5 sequential `Edit` tool calls on `server/booksRouter.ts` (364 → ~416 expected lines), the file on disk came back at 372 lines, cut off mid-statement at `getDashboardStats: protectedProcedure.query(async ({ ctx }) => ` (no body, no trailing newline). The tell: the corrupted file's byte count was **exactly 18521 bytes — identical to the pre-edit original**, despite visibly containing the new content inserted earlier in the file. That means the write silently stopped once total length reached the *original* file's byte length, dropping everything after, rather than extending the file. This confirms it's a mount-level write-path issue (this Windows-mounted folder), not a logic bug in the edit content itself. Recovered by diffing against `git show HEAD:<file>` and reconstructing the correct tail via `head`/`sed`, then writing the merged result with a plain `cat > file` redirect from bash (not the Edit/Write tools) — consistent with what prior sessions (06-16, 06-18, 06-19, 07-13) found. Re-verified with tsc + vitest after recovery, both clean. Policy going forward, restated: **do not trust Edit/Write tool output on this mount for files being grown past their current size — verify line count/byte count/tail after every such write, and fall back to bash `cat`/heredoc if it doesn't match.**
- Stale `.git/HEAD.lock`, `.git/index.lock`, and per-branch `refs/heads/*.lock` files from earlier sessions are still unremovable from the sandbox (`rm`/`chattr` both fail with `Operation not permitted`, and `lsattr` reports `Operation not supported` — this is the mount, not a permissions issue we can fix from here). Normal `git add`/`commit`/`update-ref` are blocked as a result. Workaround (same as prior sessions): stage via a scratch `GIT_INDEX_FILE`, build the tree with `write-tree`, create the commit object with `commit-tree` (this bypasses the index/HEAD locks entirely — plumbing, not porcelain), then advance the branch ref by overwriting `.git/refs/heads/<branch>` directly with the new SHA (a plain file write, not a locked ref-update). This commits cleanly to history; only the ref advances without touching HEAD.lock.
- Scoped tsc-with-tests trick still needed: root project's `types: ["node", "vite/client"]` can't resolve `vite/client` when the client is out of scope, so the scratch tsconfig used here overrides `types: ["node"]` only, plus an explicit `typeRoots` pointing at `node_modules/@types`. Full command took ~15s, well inside the 45s cap.
